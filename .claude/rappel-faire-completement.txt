RAPPEL PERMANENT (à appliquer à CHAQUE message, sans exception) :

- RÈGLE N°1 — Zéro chargement tout seul. Le seul texte que je charge automatiquement, c'est ce hook. Je n'ouvre aucun autre document de ma propre initiative. Si j'ai besoin d'en lire ou d'en écrire un, je le demande AVANT — sinon c'est l'utilisateur qui me le propose.

- RÈGLE N°4 — CRUD encadré + EnBase / Get–Put / zéro copie (à appliquer AVANT toute écriture de code). La base de données est la SEULE source de vérité ; toute donnée métier vit en base, à UN seul endroit. CRUD = les 4 seules opérations possibles sur une donnée : Create (créer), Read (lire), Update (modifier), Delete (supprimer) — et chaque opération qui ÉCRIT ou EFFACE passe D'ABORD un contrôle métier, jamais automatique : • CREATE — je refuse de créer si la donnée existe déjà (unicité) ; • UPDATE — je ne modifie pas une donnée figée par son usage (ex. facture validée = gelée par la loi ; plein de cas semblables) : je vérifie d'abord en base qu'elle est encore libre ; • DELETE — je ne supprime pas une donnée encore référencée par d'autres tables (lien / clé étrangère) ; • READ — aucune contrainte, la lecture est toujours libre. PRINCIPE D'ÉCRAN : un écran n'est qu'une fenêtre sur les tables — ses champs SONT les colonnes réelles (get pour afficher), ses boutons SONT le CRUD ; Valider = put (on écrit en base), Annuler = rien (aucune écriture, on garde la valeur lue). On ne travaille QUE comme ça : get pour lire, put pour écrire, zéro donnée en dur. Ce principe ne change pas avec la complexité : un écran sur UNE table ou une page sur PLUSIEURS tables reliées (par leurs clés étrangères) = le même geste, juste plusieurs get/put sur les bonnes tables. Un cas compliqué peut avoir PLUSIEURS boutons Valider (validation par étapes qui se dépendent — tant qu'une étape n'est pas validée on ne passe pas à la suivante, ex. création d'un référentiel) : chaque Valider reste un put, chaque étape lit/écrit ses bonnes tables. ZÉRO COPIE : get = je LIS la donnée en base chaque fois que j'en ai besoin ; put = j'ÉCRIS la donnée en base quand elle change. INTERDIT : recopier une donnée qui existe déjà ailleurs (colonne en double, variable qui la retient, fichier, cache, « métadonnée ») — une copie finit TOUJOURS par diverger de la source et mentir. BARRIÈRE obligatoire avant d'ajouter le moindre champ / colonne / variable / fichier qui STOCKE une donnée : (1) cette donnée existe-t-elle déjà en base ? → si oui, je la LIS (get), je ne la recopie jamais ; (2) ce que je m'apprête à écrire, quelqu'un le LIT-il ? → si personne ne le lit, je ne l'écris pas ; (3) une donnée = une seule place, get pour lire, put pour écrire, zéro copie. En cas de doute sur où ranger une donnée : je DEMANDE avant de créer quoi que ce soit.
