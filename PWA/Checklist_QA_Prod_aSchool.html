Checklist QA Production PWA aSchool

Objectif

Valider les points critiques avant mise en ligne afin d'éviter la majorité des incidents réels. Passer sur mobile physique (pas émulateur DevTools) — iPhone ou Android réel.

Comment utiliser cette checklist — scénario réel

Durée : 25-35 minutes. 1 seul testeur. iPhone dans la main.

  1. Ouvrir ce fichier sur un écran, téléphone dans l'autre main.
  2. Suivre les 10 points prioritaires dans l'ordre — ils couvrent 80% des risques réels.
  3. Compléter les 21 points si le temps le permet.
  4. Remplir la colonne "Testé le / Par" sur chaque ligne cochée.

Ce que cette checklist attrape que les tests unitaires ne voient pas : le 404 sur refresh de sous-page, le Back button après logout, la bannière iOS absente, le second onglet resté connecté.

Règle : un point rouge bloquant = pas de déploiement. Un point cosmétique (icône légèrement floue, couleur splash) = non bloquant, noter et continuer.

Détail des points

1. Connexion navigateur normal Login + refresh de page + session persistante. Pourquoi : valide cookies, auth backend, CSRF.

2. Connexion app installée (PWA) Installer puis se connecter depuis l'icône écran d'accueil. Pourquoi : le mode standalone peut gérer différemment les cookies httpOnly.

3. Déconnexion réelle — 3 sous-tests obligatoires

  • Logout → relancer l'app → ne pas être reconnecté automatiquement
  • Logout → bouton Back du navigateur → ne pas revenir sur une page protégée
  • Logout dans l'onglet A → l'onglet B redirige automatiquement vers /login

Pourquoi : sessions fantômes, cache UI, historique navigateur, onglets orphelins.

4. Installation Android Vérifier bannière d'installation native + lancement depuis écran d'accueil. Pourquoi : manifeste, service worker, HTTPS, icônes.

5. Installation iPhone + bannière IOSInstallBanner

  • Ouvrir le dashboard connecté → attendre 5 secondes → la bannière bordeaux "Installer aSchool sur iPhone" apparaît
  • Cliquer ✕ → la bannière disparaît et ne réapparaît plus (même après reload)
  • Safari > Partager > Ajouter à l'écran d'accueil → app installée et fonctionnelle

Pourquoi : iOS n'a pas de popup install automatique — la bannière est le seul vecteur d'installation.

6. Icône propre Vérifier netteté sur écran d'accueil Android et iOS. Pourquoi : image 192/512 correcte, pas de flou ni de bords coupés.

7. Splash / thème couleur Ouvrir l'app installée depuis l'écran d'accueil. Pourquoi : vérifier theme_color bordeaux, background_color blanc, meta apple-mobile-web-app-capable.

8. Refresh sur sous-page Naviguer vers /mes-activites ou /profil puis F5 ou pull-to-refresh mobile. Pourquoi : éviter 404 React Router — le SW doit renvoyer index.html.

9. API après installation Créer, modifier, sauvegarder des données depuis l'app installée. Pourquoi : vérifier que le SW ne met pas en cache /api/ (NetworkOnly configuré).

10. Mode hors ligne + OfflineBanner

  • Couper réseau → relancer l'app → shell visible sans crash
  • La bannière bordeaux "Hors connexion" apparaît en haut de page
  • Tenter une génération → message d'erreur propre, pas de crash silencieux

Pourquoi : UX offline dégradée mais lisible plutôt qu'écran blanc.

11. Réseau lent + message timeout Simuler 3G lente (DevTools > Network > Slow 3G). Déclencher une génération → attendre → le message "Connexion lente ou indisponible. Vérifiez votre réseau et réessayez." apparaît après timeout. Pourquoi : fetchWithTimeout actif — vérifier que le message remonte bien dans l'UI.

12. Mise à jour version + UpdateBanner

  • Déployer une nouvelle build
  • Ouvrir l'app depuis une session existante → la bannière "Nouvelle version disponible" apparaît
  • Cliquer "Actualiser" → la page recharge sur la nouvelle version
  • Vérifier qu'aucune ancienne session ne reste bloquée sur l'ancienne version

Pourquoi : skipWaiting + clientsClaim actifs — le SW doit prendre la main immédiatement.

13. Responsive mobile réel Tester sur iPhone et Android : login, dashboard, formulaire de génération, Mes activités, tableaux admin. Pourquoi : usage mobile réel, pas émulateur — les vrais profs utilisent leur téléphone.

14. Clavier mobile Saisir du texte dans les champs principaux sur iPhone et Android. Vérifier que les boutons d'action restent visibles quand le clavier est ouvert. Pourquoi : le clavier mobile réduit la hauteur visible, les boutons bottom-right peuvent disparaître.

15. HTTPS strict Vérifier cadenas dans la barre d'adresse, aucune alerte mixed content en console. Pourquoi : requis pour PWA + cookies Secure + SameSite.

16. Multi-onglets + cross-tab logout

  • Ouvrir deux onglets sur aSchool
  • Se déconnecter dans l'onglet A
  • Vérifier que l'onglet B redirige automatiquement vers /login sans action manuelle

Pourquoi : sans listener storage, les onglets orphelins restent connectés indéfiniment.

17. Taille bundle DevTools > Lighthouse > Performance + PWA. Vérifier score PWA ≥ 90. Vérifier Network > JS bundle < 500 Ko gzippé. Pourquoi : performances mobile sur réseau moyen, critère de qualité PWA.

18. Cache clean logout Logout → passer en mode hors ligne → rouvrir l'app. Aucune donnée utilisateur visible, aucune page protégée accessible. Pourquoi : vérifier que localStorage et sessionStorage sont bien vidés au logout.

19. Comptes réels / rôles Tester avec un compte enseignant réel + un compte admin. Vérifier que chaque rôle voit uniquement ce qu'il doit voir. Pourquoi : les permissions sont souvent spécifiques par rôle — un admin ne doit pas bloquer un enseignant et vice-versa.

20. Console + logs prod

  • Console navigateur : aucune erreur rouge après chaque action testée
  • Logs Nginx VPS : aucun 5xx après les actions (génération, login, save)

Pourquoi : détecter les erreurs silencieuses côté client et les crashs backend non affichés.

21. DevTools Application — SW + manifest DevTools > Application :

  • Service Workers → statut "activated and running", pas de "waiting"
  • Manifest → aucune erreur, icônes résolues, start_url correcte
  • Cache Storage → vérifier que /api/ n'apparaît PAS dans les caches listés

Pourquoi : 30% des bugs PWA silencieux sont détectables ici en 2 minutes.

Priorité si seulement 10 minutes

  1. Login app installée
  2. Déconnexion — les 3 sous-tests
  3. Refresh sous-page
  4. Install Android
  5. Install iPhone + bannière
  6. API create/save
  7. Offline + OfflineBanner
  8. Update build + UpdateBanner
  9. Multi-onglets cross-tab logout
  10. DevTools Application (SW + manifest)

Verdict

Si tous les points passent, le lancement est fortement sécurisé. Un point rouge = bloquant si lié à auth, données, ou install. Cosmétique = non bloquant.
